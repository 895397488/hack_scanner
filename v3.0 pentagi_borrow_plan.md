## 第一阶段：核心代理架构升级（P0 — 最优先）

### 1.1 Multi-Agent 框架

**PentAGI 的核心价值：** 不是单个扫描器，而是 Researcher → Developer → Executor 的协作链。Researcher先调研目标技术栈和攻击面，Developer设计测试计划，Executor执行测试并反馈结果。

**借鉴方案 — 新增 `agents/` 包：**

```
hack_scanner.py          # CLI入口（保持不变）
+ agents/                # 新建
│   ├── orchestrator.py   # Agent编排器：解析任务→拆分子任务→分派Agent
│   ├── researcher.py     # Researcher Agent：信息收集、技术栈分析、攻击面枚举
│   ├── developer.py      # Developer Agent：测试计划制定、payload设计
│   ├── executor.py       # Executor Agent：执行扫描、收集结果、反馈
│   └── memory.py         # Agent间共享记忆（上下文传递）
```

**关键设计：**
- 不做 Go/微服务，保持 Python
- Researcher 负责：目标侦察、API端点发现、技术栈分析、敏感文件定位
- Developer 负责：基于Researcher结果制定测试策略和针对性payload
- Executor 负责：执行具体扫描模块（SQLi/XSS/SSRF等）并汇总结果
- Agent间通过 `memory.py` 传递上下文，避免重复工作

**PentAGI 的监督机制也要借鉴：**
- **Mentor模式**：当连续5次相同工具调用或10次总调用无进展时自动介入，建议新策略
- **Reflector纠错**：LLM无法生成有效工具调用时，Reflector分析失败原因并给出指导

### 1.2 Agent Prompt 增强

当前 AI prompt 过于通用。借鉴 PentAGI 的 `system/human/tool prompt templates`：

```python
# agents/prompts/

```

## 第二阶段：记忆与知识系统（P1）

### 2.1 File-Based Memory System

**不用 PostgreSQL/Neo4j，但借鉴其设计理念：**

```
memory/                     # 新建目录
├── episodic/              # 事件记忆（每次扫描的结果）
│   └── 2026-06-22_1430.json  # 按日期+时间存储
├── semantic/              # 语义记忆（可复用的知识和pattern）
│   ├── sqli_patterns.json     # SQL注入模式库
│   ├── xss_bypasses.json      # XSS绕过技巧
│   └── common_vulns.json      # 常见漏洞配置映射
├── working/               # 工作记忆（当前任务上下文）
│   └── current_target.json
└── index.md               # 内存索引（类似RAG的向量搜索替代方案）
```

**借鉴 PentAGI 的三层记忆模型：**
- **Long-term Memory** → `semantic/`：扫描中发现的可复用知识
- **Working Memory** → `working/`：当前扫描任务的上下文
- **Episodic Memory** → `episodic/`：每次扫描的历史记录

### 2.2 Search Engine Integration

PentAGI 集成了 DDG/GT/Tavily/Traversaal/Perplexity/Searxng。我们不需要全量，但至少要集成 **Tavily + DuckDuckGo**（已有 `tavily-search` skill）：

```python
# search_engines.py  新增
├── duckduckgo_search.py   # DDG免费搜索
├── tavily_search.py       # Tavily API（已有skill）
├── searxng_search.py      # SearXNG元搜索（可选，本地部署）
└── sploitus_search.py     # 漏洞数据库搜索（sploitus.com）
```

**用途：** 
- 扫描时自动搜索目标相关的已知漏洞和exploit
- AI分析结果时作为补充情报源
- 发现CVE时自动匹配exploit代码

## 第三阶段：API与外部集成（P2）

### 3.1 REST API Server

借鉴 PentAGI 的 Flask/FastAPI REST + GraphQL，我们先用 **FastAPI** 提供基础API：

```python
# api/                     # 新增
├── app.py                # FastAPI应用
├── routes/
│   ├── scans.py          # /api/scans - 扫描管理
│   ├── reports.py        # /api/reports - 报告下载
│   └── agents.py         # /api/agents - Agent控制
├── models.py             # Pydantic模型定义
└── auth.py               # API Key鉴权（Bearer Token）
```

**核心端点：**
- `POST /api/scans` — 创建新扫描（URL或文件路径）
- `GET /api/scans/{id}` — 查询扫描状态和结果
- `DELETE /api/scans/{id}` — 取消扫描
- `GET /api/reports/{id}/html` — 下载HTML报告
- `GET /api/reports/{id}/json` — 下载JSON数据
- `POST /api/agents/control` — Agent操作（暂停/继续/修改策略）

### 3.2 WebSocket 实时进度

借鉴 PentAGI 的 GraphQL Subscription + WebSocket，提供实时扫描进度推送：

```python
@router.websocket("/ws/scans/{scan_id}")
async def scan_progress(websocket: WebSocket, scan_id: str):
    """实时推送扫描进度"""
    await websocket.accept()
    async for event in scan_events(scan_id):
        await websocket.send_json(event)
    # {type: "finding", data: {...}}
    # {type: "status", data: {"progress": 67, "current_phase": "sql_injection"}}
    # {type: "complete", data: {"findings_count": 15}}
```

## 不借鉴的部分

1. **Go语言后端** — 保持Python，降低门槛
2. **React前端** — 当前HTML报告够用，后续可考虑 Streamlit/Gradio
3. **GraphQL** — REST更简单直接
4. **Neo4j/Knowledge Graph** — 对扫描工具来说过度工程化
5. **Grafana/Jaeger/Langfuse全栈监控** — 对独立工具太重
6. **Docker Compose全栈编排** — 当前纯Python的 simplicity 是核心优势
